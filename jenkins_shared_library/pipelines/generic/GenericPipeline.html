Package: jenkins_shared_library.pipelines.generic

[Groovy] Class GenericPipeline

      • jenkins_shared_library.pipelines.generic.GenericPipeline

      • Extends the functionality available in the Pipeline class. This class adds methods for building and testing your application.

        A typical pipeline should include these stage in sequence:

        • build : build and generate artifact locally.
        • test : test the artifact.
        • sonarScan : SonarQube static code scanning.
        • packaging : package/create artifact.
        • publish : publish the artifact to Artifactory. If the build is a release, we publish the artifact to release folder.
        • release : if the build is a release, we tag the GitHub. If the build is a formal release, we also bump release on code base.

        Required Plugins:
        The following plugins are required:
        • All plugins listed at Pipeline
        • Credentials Binding
        • JUnit
        • HTML Publisher
        • Cobertura
        Example:
         @Library('fill this out according to your setup') import org.zowe.jenkins_shared_library.pipelines.generic.GenericPipeline
 node('pipeline-node') {
     GenericPipeline pipeline = new GenericPipeline(this)

     // Set your config up before calling setup
     pipeline.admins.add("userid1", "userid2", "userid3")

     // update branches settings if we have different settings from #defineDefaultBranches()
     pipeline.branches.addMap([
         [
             name         : 'lts-incremental',
             isProtected  : true,
             buildHistory : 20,
         ]
     ])

     // MUST BE CALLED FIRST
     pipeline.setup(
         // Define the git configuration
         github: [
             email: 'git-user-email@example.com',
             usernamePasswordCredential: 'git-user-credentials-id'
         ],
         // Define the artifactory configuration
         artifactory: [
             url : 'https://your-artifactory-url',
             usernamePasswordCredential : 'artifactory-credential-id',
         ]
     )

     pipeline.build()   // Provide required parameters in your pipeline
     pipeline.test()    // Provide required parameters in your pipeline
     pipeline.publish() // Provide required parameters in your pipeline
     pipeline.release() // Provide required parameters in your pipeline

     // MUST BE CALLED LAST
     pipeline.end()
 }

          • java.lang.String artifactoryUploadTargetFile

            Artifactory file name pattern when the pipeline try to publish artifacts.

            Note:
            Allowed macros are same as macros defined in artifactoryUploadTargetPath and with these extras.
            • - filename: the base file name extracted from original artifact.
            • - fileext: the file extension extracted from original artifact.
            • - publishversion: parsed string of publishTargetVersion

            Default:
            "{filename}-{publishversion}{fileext}"
            Example:
            With the default pattern, if we want to upload my-artifact.zip, the pipeline may interpret the targe file name as "my-artifact-1.2.3-snapshot-23-20190101010101.zip".

          • java.lang.String artifactoryUploadTargetPath

            Artifactory upload path pattern when the pipeline try to publish artifacts.

            Note:
            Allowed macros:
            • - repository: Artifactory repository name where the artifact will be published.
            • - package: value defined by pipeline.setPackage(name)
            • - subproject: optional value passed when parsing the path string
            • - version: the current version
            • - prerelease: pre-release string, like rc1, beta1, etc.
            • - branchtag: branch tag
            • - branchtag-uc: branch tag but in UPPER case.
            • - timestamp: timestamp in yyyyMMddHHmmss format
            • - buildnumber: current build number
            Note:
            If the artifactory repository is a Maven repository, the upload may fail if the branchtag is lower case characters.
            Default:
            "{repository}/{package}{subproject}/{version}{branchtag-uc}/"
            Example:
            With the default pattern, the pipeline may interpret the targe upload path as "lib-snapshot-local/org/zowe/my-project/1.2.3-STAGINGS-TLS/".

          • void releaseGeneric(ReleaseStageArguments arguments)

            Creates a stage that will execute a release

            The default behavior of release stage is after the publish stage, which release artifacts have been uploaded to Artifactory, we have to tag the GitHub branch with the release. If this is a formal release, we also need to bump project version, so we won't build same version again. The version bump will create a commit and push to GitHub. After all, the stage will send out email notification for the new release.

            Provide arguments.operation to override the default behavior. Or you can provide arguments.tagBranch or arguments.bumpVersion to override part of the behavior.

            Calling this function will add the following stage to your Jenkins pipeline. Arguments passed to this function will map to the org.zowe.jenkins_shared_library.pipelines.generic.arguments.ReleaseStageArguments class. The org.zowe.jenkins_shared_library.pipelines.generic.arguments.ReleaseStageArguments#operation will be executed after all checks are complete. This must be provided or a java.lang.NullPointerException will be encountered.

            Stages:
            This method adds the following stage to your build:
            Releasing: org.zowe.jenkins_shared_library.pipelines.generic.arguments.ReleaseStageArguments#name
            This stage is responsible for tagging the branch and bumping the release of your application source.
            Conditions:

            This stage will adhere to the following conditions:

            • The stage will only execute if the current build result is org.zowe.jenkins_shared_library.pipelines.base.enums.ResultEnum#SUCCESS or higher.
            • The stage will only execute if the current branch is a releasing branch.
            • The build job is started when build parameter "Perform Release" BUILD_PARAMETER_PERFORM_RELEASE is checked.

            Exceptions:

            The following exceptions will be thrown if there is an error.

            org.zowe.jenkins_shared_library.pipelines.generic.exceptions.ReleaseStageException
            When stage is provided as an argument.
            When publish stage is not successful.
            java.lang.NullPointerException
            When an operation is not provided for the stage.

            Note:
            This method was intended to be called release but had to be named releaseGeneric due to the issues described in Pipeline.

            Creates a stage that will perform SonarQube static code scanning.

            The default behavior of this stage will check if there is a "sonar-project.properties" file presents. If so, then run a SonarQube static code scanning using that configuration file.

            Provide arguments.operation to override the default behavior.

            Calling this function will add the following stage to your Jenkins pipeline. Arguments passed to this function will map to the SonarScanStageArguments class. The SonarScanStageArguments.operation will be executed after all checks are complete. This must be provided or a java.lang.NullPointerException will be encountered.

            Stages:
            This method adds the following stage to your build:
            SonarQube Scan: SonarScanStageArguments.name
            Perform a SonarQube static code scanning on your application. The stage also ignores any SonarScanStageArguments.resultThreshold provided and only runs on org.zowe.jenkins_shared_library.pipelines.base.enums.ResultEnum#SUCCESS.

            Exceptions:

            The following exceptions can be thrown by the sonar scan stage:

            SonarScanStageException
            When arguments.stage is provided. This is an invalid argument field for the operation.
            When called more than once in your pipeline. Only one sonar scan may be present in a pipeline.
            When arguments.scannerServer is not defined and arguments.operation is not provided.
            When arguments.scannerTool is not defined and arguments.operation is not provided.
            java.lang.NullPointerException
            When arguments.operation is not provided.

            Note:
            With default behavior (by not providing arguments.operation), the SonarQube scan stage will only run if "sonar-project.properties" file presents. This file should have basic scanning configurations.
            Note:
            With default behavior (by not providing arguments.operation), the SonarQube scan stage won't fail the build if the scan result doesn't pass the threshold. The stage will only fail when then scanning cannot be performed, like cannot connect to the SonarQube server.

          • void testGeneric(java.util.Map arguments = [:])

            Creates a stage that will execute tests on your application.

            Arguments passed to this function will map to the TestStageArguments class.

            Stages:
            This method adds the following stage to the build:
            Test: TestStageArguments.name

            Runs one of your application tests. If the test operation throws an error, that error is ignored and will be assumed to be caught in the junit processing. Some test functions may exit with a non-zero return code on a test failure but may still capture junit output. In this scenario, it is assumed that the junit report is either missing or contains failing tests. In the case that it is missing, the build will fail on this report and relevant exceptions are printed. If the junit report contains failing tests, the build will be marked as unstable and a report of failing tests can be viewed.

            The following reports can be captured:

            Test Results HTML Report
            This is an html report that contains the result of the build. The report must be defined to the method in the org.zowe.jenkins_shared_library.pipelines.generic.arguments.TestStageArguments#htmlReports variable.
            Code Coverage HTML Report
            This is an HTML report generated from code coverage output from your build. The report can be omitted by omitting org.zowe.jenkins_shared_library.pipelines.generic.arguments.TestStageArguments#htmlReports
            JUnit Report
            This report feeds Jenkins the data about the current test run. It can be used to mark a build as failed or unstable. The report location must be present in org.zowe.jenkins_shared_library.pipelines.generic.arguments.TestStageArguments#junit
            Cobertura Report
            This report feeds Jenkins the data about the coverage results for the current test run. If no Cobertura options are passed, then no coverage data will be collected. For more information, see org.zowe.jenkins_shared_library.pipelines.generic.arguments.TestStageArguments#cobertura

            The test stage will execute by default if the current build result is greater than or equal to org.zowe.jenkins_shared_library.pipelines.base.enums.ResultEnum#UNSTABLE. If a different status is passed, that will take precedent.

            After the test is complete, the stage will continue to collect the JUnit Report and the Test Results HTML Report. The stage will fail if either of those are missing. If specified, the Cobertura Report are then captured. The build will fail if these reports are to be collected and were missing.

            Exceptions:

            The test stage can throw the following exceptions:

            org.zowe.jenkins_shared_library.pipelines.generic.exceptions.TestStageException
            When a test stage is created before a call to buildGeneric(Map)
            When TestStageArguments.junit is missing
            When invalid options are specified for TestStageArguments.junit
            When TestStageArguments.cobertura is provided but has an invalid format
            When TestStageArguments.operation is missing.

            Note:
            This method was intended to be called test but had to be named testGeneric due to the issues described in Pipeline.

            Parameters:
            arguments - A map of arguments to be applied to the TestStageArguments used to define the stage.
